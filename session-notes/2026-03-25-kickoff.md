# 2026-03-25 — Project Kickoff

## What Happened

Started from exploring Claude Code's new **channels feature** (research preview, v2.1.80+). Channels let MCP servers push events into a running Claude Code session — the inverse of normal MCP where Claude queries on-demand. Supported channels: Telegram, Discord, iMessage, and a localhost demo (fakechat).

**Key constraint discovered**: Channels require a claude.ai login. Since Avi accesses Claude through a proxy server with a localhost API key, channels are off the table entirely.

This led to looking at **Bawa's [inline-claude](https://github.com/bawakul/inline-claude)** — an Obsidian plugin that enables inline conversations with Claude Code directly within notes. Architecture: `Obsidian → Plugin → Channel Server (localhost:4321, Bun) → Claude Code → back`. It uses channels under the hood, so same login constraint.

The real question became: **what's the simplest architecture that works without channels?**

### The Answer: `claude -p` as Backend

`claude -p` (non-interactive mode) works through the proxy. It also gets MCP tool access. So the architecture is:

```
Any Frontend → Local HTTP Server → claude -p (with MCP tools) → Response back
```

No persistent session needed. No login. Just a thin wrapper.

### The Mental Model: Jupyter Notebooks for Prose

The core insight: this is like Jupyter Notebooks but for prose + AI. You write markdown, trigger a query inline, and the response appears as a callout block in the same document. The whole document is the context. No separate chat window, no copy-pasting.

### Model-Agnostic by Design

The server doesn't know or care which model sits behind the CLI. `claude -p`, `gemini`, Codex CLI, `ollama` — swap a single adapter function. Auth is simply "you have access to whichever CLI you're wrapping."

### Frontends Are Independent Problems

| Surface | Approach | Priority |
|---------|----------|----------|
| Obsidian | Adapt Bawa's inline-claude, swap channel server for HTTP backend | First |
| VS Code | Extension that POSTs to same endpoint | Later |
| Browser/Confluence | Browser extension injecting UI | Later |

### Paths Considered and Rejected

1. **Claude Code + Channels** — Requires login, persistent session. Won't work with proxy.
2. **Claude API directly** — Loses MCP tools, would need to reimplement integrations.
3. **Claude Code SDK (Agent SDK)** — More complex than needed for one-shot queries.

## Key Decisions

- **Model-agnostic from day one** — the server is a thin adapter, not Claude-specific
- **One-shot queries** — no conversation continuity, document context is enough
- **Obsidian first** — adapt Bawa's inline-claude as reference frontend
- **Python backend likely** (FastAPI) unless Bun proves simpler
- **Response format**: Collapsible callout blocks in markdown (Bawa's pattern)

## Open Questions

- How much document context to send? Token limits on long docs?
- Security: localhost-only binding sufficient to protect proxy API key?
